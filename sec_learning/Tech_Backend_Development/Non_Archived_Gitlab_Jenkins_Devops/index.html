部署前言

部署目标

    最近趁着过年在家里部署了私有云的基础设施。接下来基于搭建的基础设施，我计划继续部署基于 Gitlab 和 Jenkins 的 CI/CD 流水线环境，以实现自动化构建和部署流程，简化后续项目的部署压力。简单来说，部署目标如下图所示:

    首先，我会用一个 Nginx 来作为整个 CI/CD 流水线的代理网关，藏在 Nginx 后面的有 Gitlab 仓库，Jenkins，静态网站资源，以及一些后端服务。当开发人员向 Gitlab 的仓库推送修改时，Gitlab 会基于 Webhook 的设置向 Jenkins 发起请求，Jenkins 收到请求之后就会启动构建步骤:

  1. 如果开发人员向仓库推送的是静态资源，那么 Jenkins 就会将静态资源从仓库中拉取到相应的部署位置，这样外部用户通过 Nginx 代理网关就能访问到这些静态资源;
  2. 如果开发人员向仓库推送的是源代码，那么 Jenkins 就会将源代码拉取到本地，启动构建，然后把构建完成的 Docker 镜像推送到本地的镜像仓库中去，然后再从本地拉取镜像，启动容器以完成部署;

    在本文中，我将演示如何搭建这样一套机制在私有服务器上，并且基于这样一套机制部署我的个人博客网站。

部署背景

    在部署之前，我在家里的网关的地方做了基于 Openwrt 的软路由处理，并且为家中申请了公网 IP 出口。这样以来就可以在网关的位置很方面的做端口映射和防火墙等设置，这部分在本文中不会阐述，默认读者具备设置能力。

    另外，我还从腾讯云购买了一个域名，并且在家中服务器上部署脚本，实现了动态的域名解析效果，也即 DDNS。具体可以看我的另一篇文章 如何通过 DNSPod API 来实现 DDNS。本文同样默认读者已经完成了设置。
