Introduction

思路

System Paper 方法论

    系统论文是从场景的角度出发的，比如测量平台性能限制、数据中心网络流的特征等，然后探讨 在特定场景下的系统的整体设计，论文的 Problem 或者说 Motivation 都是在一定的场景下提出来的，并且一定要有强大的 Motivation Experiment 进行支撑。在有了具体场景下的 Problem 后，我们应该提出评估这个场景下系统运行的 Metric，以方便我们行文进行比较和讨论 (i.e. 牺牲了什么，换取了什么)。有了场景和 Metric 后，通常会基于工程设计直觉，提出 System Design 的 Overview，也就是最上层的系统设计，提出整个系统应该包含什么模块。然后就进入每个模块的具体设计环节，在这其中可能会涉及到对相关算法的优化。比如系统需求是一个找 Top-k Heavy Hitter 的模块，并且有精度和复杂度的要求，那么就需要根据自己本身的系统设计将已有算法嵌入其中，如果算法有不合适于当前系统的地方，就需要进行优化。因此必须意识到算法在 System 的人看来，实际上是秉承拿来主义，实在需要优化了再进行优化，而不是一开始就基于算法入手来思考如何优化，进而整个系统的设计，一定要避免这个误区。对于技术的追求，也应该时刻注意这个误区。

网络测量论文套路

    基于上面的思路，我们可以把网络测量的论文基本的内容分为两部分：算法和系统。算法也分在线算法和离线算法，在大部分的测量论文中讨论的都是在线算法。这些算法各自都是面向流数据的一个或者多个特征 (i.e. Cardinality, Heavy Hitter, Frequency)，并且有着不同的时间、空间复杂度和精确度。如上所述，我们应该把这些算法合集当作工具箱来使用，所以我会在阅读论文的过程中逐渐提取这些 本质算法，并且补充在这个系列的其他博客中，但是一定要秉承 ToolBox 的出发点！

    对于系统来说，我们一定是基于数据中心网络的某种场景，比如测量平台性能限制、数据中心网络流的特征等，这也是我们论文的出发点。而在论文在初始 Design Overview 后，模块设计会涉及到这些算法的使用。应该秉承一个 "先把这些经典算法套进去模块试试看" 的思路，然后再逐渐对算法的各方面效率进行优化。

Metrics

    测量系统的 Evaluation 通常参考以下几个 Metrics:

    Generic: 测量结构是否支持多种不同的测量任务。

    Latency: 测量结构 Update 和 Query 的延迟。

    Accuracy: 估测一类或者多类流量特征的准确度。

    CPU Usage: 每包的处理所需要的计算资源的消耗。

    Memory Usage: 测量结构所需占用的内存大小。由于测量结构被预加载到 Cache 中是被期望的，所以即使在服务器内存资源充足的情况下也仍需要考虑测量所需要占用的内存空间大小。

    Convergence Time: 整个测量过程需要花费多久才能达到预期的测量精度。

Frequency Estimation

    数学形式表达:

Frequency Estimation Problem: 给定一个由 `n` 个元素 `t_1`, `t_2`, `...`, `t_n` 组成的流 `S`。设流中各个元素 `t_j` 属于种类 `i`，定义种类 `i` 出现的频率为 `f_i=|{j|t_j=i}|`。该问题的目标是对各种元素种类 `i` 的出现频率 `f_i` 进行估计，得到 `\hat{f_i}`。`\hat{f_i}` 需要满足关系 `\hat{f_i} \leq f_i \leq \hat{f_i}+\epsilonn`，其中 `\epsilon` 是一个相对误差系数。

测量系统: 针对流量特征的优化

总结近几年来的关于网络测量系统的论文

[SIGCOMM'18] Elastic Sketchelasticsketch

Motivation

    这篇文章面向的场景是: 数据中心网络中动态的流量特征 (i.e. 动态带宽、动态到达率和动态流量分布) 下的网络测量问题，具体如下:

  1. 动态带宽: 当数据中心的网络变得拥塞的时候，多测量节点的 Sketch 传给 Controller 的这个过程不应该成为网络的负担，因此需要考虑测量结构的压缩;
  2. 动态到达率: 当数据包的到达率快速上升的时候，测量结构如何对数据到达率的动态适应，以重点记录大流的信息?
  3. 动态流量分布: 流量中有大流和小流，straightforward 的测量方法是把大流和小流分开测，但是大流和小流的数量可能会发生动态变化，也即分布是动态的，分开测的话，现有的测量方法没有办法动态地调整大流和小流测量结构的大小

Design

    如上所示是 Elastic Sketch 的基本设计。Overview 来说就是其将大流用一个 Heavy Part 的结构挡在了 CM-Sketch 之外。Heavy Part 是一个 Counter-based 的设计，并且有一个驱逐的过程。加入一条流的数据从 Heavy Part 中被驱逐出来了，那么它就会被送入 CM-Sketch 中去。

    针对动态带宽，文章提出了针对 Light-part 的压缩算法; 针对动态到达率，当到达率上升时，只有 Heavy Part 会被更新; 针对动态流量分布，文章提出了 Heavy Part 进行自我复制以扩展 Heavy Part 哈希桶数量的方法。

测量系统: 针对测量平台性能限制的优化

[SIGCOMM'19] Nitro Sketchnitrosketch

Motivation

    Sketch 被用于网络测量的初衷是因为它能够使用非线性大小的内存区域，在一定的误差范围内完成对网络流的测量。但是在软件交换机上:

  1. 内存并不是最紧缺的资源: 内存容量在软件交换机上一般都十分可观;
  2. 内存并不是带宽的瓶颈: 一般来说，Sketch 的容量足以被加载到 LLC 中去，因此访存速度可以得到保证

    这篇文章的实验表明，Sketch 算法中针对每个数据包的多次哈希计算才是导致 Sketch 方案在部署过程中单核测量性能过低的原因。

Design

    NS 的思路是: 每当一个包到达时，对要更新的 Hash 表进行抽样，并且只更新抽中的哈希表，以此来避免针对每包的多次哈希计算操作。并且为了避免对每个哈希表都进行一次 coin-flip (i.e. 这种计算很昂贵)，NS 计算的是下一个要更新的哈希表的距离，这个距离随机地符合几何分布，并且可以动态随着 packet 的到达率进行调整。

[SIGCOMM'17] Sketchvisorsketchvisor

Motivation

    这篇文章瞄准的问题是 Sketch 在实际部署过程中的 CPU 开销问题。如上图所示，文章的实验首先说明了四种 Sketch 系统 ( univmon, flowradar, deltoid, revsketch ) 需要为每个数据包付出的 CPU 运算周期代价；其次文章发现这四种 Sketch 的单核性能均不超过 5Gbps，其中 deltoid 的五核性能甚至也不超过 5Gbps。文章通过分解 CPU 运行周期发现这几种方案的大部分时间都花费在了哈希计算上，因此这是一个急需解决的瓶颈。

    一种比较简单直接的方法是不使用像 Sketch 这样的多哈希表结构，转而采用单哈希表结构来进行测量。单哈希表可行的理由是流的分布存在一定的偏移性，结合现代的 Cache 的管理机制可以实现快速高效的测量 (i.e. 更少的 CPU 周期和更高的 Cache 命中率)，但是其花费的总的内存区域大小仍然过大。因此需要找出一种方法来使得 Sketch 在实际系统上的部署变得可行。

Insight & Challenge

    这篇文章的出发点很简单: 既然 Sketch 的处理带宽性能无法达到预期，那么就另辟蹊径。如上图所示的第一部分，首先主机会部署有传统的 Sketch 测量模块，也即 Slow Path。当 Dataplane 的数据包的到达率上升使得 Ingress Buffer 达到一定的阈值时，那么数据包将会被引导至 Fast Path 进行处理。Fast Path 的测量模块是十分轻量的，这意味着在 Fast Path 的测量所产生的计算和访存开销是很小的，但是这也同时意味着 Fast Path 将会引入较大的测量误差。为了解决 Fast Path 带来的误差，这篇文章又从 Control Plane 入手，在控制平面上收集各个 Host 的测量信息 (i.e. 包括 Slow 和 Fast Path) 的全局测量信息，然后基于一定的算法尽量消除由 Fast Path 引入的测量误差，因此这篇文章的设计内容实际上包括两个:

  1. 基于 Counter 的 Fast Path 数据面测量模块设计;
  2. 基于压缩感知 (Compressive-sensing) 的控制面恢复算法;

Design

    对于上述的第一点 Fast Path 的测量模块，这篇文章实际上维护了一个哈希表来记录 Top-k 流。在 Fast Path 只记录 Top-k 流的原因是因为文章相信数据中心内部流量的大部分是由少数大流主导的，也即数据中心流量呈现长尾分布 (Heavy-tailed Pattern)，因此如果发生高的数据包到达率使得系统切换到 Fast Path，那么这些流量中的很大一部分事实上是少数的几个大流，所以文章认为在 Fast Path 中记录 Top-k 大流是足够的。在具体设计上，文章基于 Misra Gries 算法，结合 Probabilistic Lossy Counting (PLC) 算法 plc，设计了用于追踪 Top-k 大流频率的算法。

    对于第二点，文章从 Network-wide 的角度出发，在收集完整个网络的 Sketch 以及 Fast Path 的 Top-k Heap 后，进行数据的融和和计算。这部分文章将问题建模称矩阵的差值问题，并且使用基于压缩感知的方法来进行记录数据的恢复。
