    现在我们可以简单地分析出日志如此打印的原因: Line 1 的打印发生在 test_func 中向 vector 中 push_back 新元素时，我们实例化了一个新的 Obj 对象; Line 2 的打印发生在 push_back 内部，对新创建的 Obj 对象进行了一次拷贝，因此执行了一次拷贝构造函数; Line 3 的打印显示了 test_func 函数的结束；重点是 Line 4 的打印，其打印的原因是我们在 main 中将 test_func 返回的 vector 对象 复制 给了变量 k，因此此时又会执行一次拷贝构造函数。

    由于对象的拷贝构造的开销是非常大的，因此在开发过程中，可以避免的话就尽可能避免。在上面的两次拷贝构造中，第一次是 vector 的特性决定的，不可避免；而第二次拷贝构造，在 C++11 中就是可以避免的了。我们在使用 g++ 进行编译时，加上 -std=c++11 编译参数，程序的运行输出就变成了:

    可以发现，当我们指定使用 C++11 标准来编译程序时，原先出现在 Line 4 的第二次拷贝构造就消失了。下面我们简单分析前后两个版本程序的区别。在老版本的编译中:

  1. test_func 返回一个临时的 vector 对象 (我们这里使用 tmp 来标识它);
  2. 执行 vector 的 = 操作符函数重载函数，将对象 k 中原有的容器元素删除，将 tmp 的容器元素复制到 k 中;
  3. 删除临时对象 tmp;

    在 C++11 版本中，执行过程如下:

  1. test_func 返回一个临时的 vector 对象 (我们这里使用 tmp 来标识它);
  2. 执行 vector 的 = 操作符函数重载函数，将对象 k 中原有的容器元素和 tmp 中的容器元素互换，此时 k 中包含的元素就被替换为了 tmp 中原有的元素;
  3. 删除临时对象 tmp (此时就删除了原有的 k 中的元素);

    细心的读者已经发现，关键在于第 2 步: C++11 版本的做法是交换而不是复制，从而避免了成员的拷贝，从而提升了程序的运行性能。因此我们也总结出了优化规律: 对于 = 右边是一个临时变量的情况，由于临时变量马上会被运行时删除掉，因此此时交换和复制的效果是一样的，可以使用交换来替代复制，以提高性能。

移动复制函数

    为了解决上述的问题，C++ 中引入了一个新的 = 操作符函数的重载形式:

    这个赋值函数通常称为 移动赋值函数，和之前版本的赋值函数相比，它有两点区别:

  • 入参不是 const 修饰的，因此它是可以对入参进行修改的;
  • 入参前有两个 & 号，这是 C++11 引入的新语法，称为 右值引用 (Rvalue Reference)，它的使用方式与普通引用是一样的，唯一的区别是可以指向临时变量;

    因此，我们可以定义两个版本的赋值函数，C++11 在编译时也会自动进行适应:

  • 如果入参是临时变量，则执行移动赋值函数，如果没有定义移动赋值函数，则执行复制赋值函数 (以兼容老版本代码);
  • 如果入参不是临时变量，则执行普通的复制赋值函数;
