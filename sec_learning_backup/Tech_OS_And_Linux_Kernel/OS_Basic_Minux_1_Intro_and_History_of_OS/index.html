0. 写在前面：从学习操作系统看学习系统设计的哲学

    当我们讨论操作系统的时候，一开始我们会发自内心自觉对其十分了解，但是当我们尝试去深入理解它的工作原理时，我们又会发现它是如此深奥、复杂和界限模糊的一种软件，因此我们常常会回过头来看操作系统它究竟在系统中扮演了一个什么样的角色，这个过程是一直在不断迭代的。在本篇文章中我们就先将操作系统的角色以可能比较抽象的语言进行定义和描述，并且对操作系统的历史进行一下回顾，这种抽象的理解大概率会是正确的（越是抽象的理解就越不会犯错误），并且我猜测随着读者（我也是读者）对操作系统的理解逐渐加深，这种抽象程度会逐渐减弱，越来越具像化到具体的代码实现上去。我的思路是：学习一种东西不能一直浮游于概念的理解，必须尝试动手自己实现，才能够从创造这个东西的人们的角度出发去思考问题的本质，从而对这个被创造出来的东西有本质性的自己的理解，尤其是像操作系统这样庞大、历史悠久、复杂的系统，更应如此。下面，我们就将开始我们对操作系统逐步的剖析旅途。

1. 操作系统的角色与功能

(1) OS 在系统中的位置：上面是什么？下面又是什么？

    如上图所示，在计算机系统中，我们在底层固然有一堆 物理硬件，硬件是整个计算机系统的骨架和物理载体，若要进行细分，可以从最底层的物理开始分起，到模拟器件，再到数字器件，再到集成电路芯片等，这超出了本系列文章的描述范围，不再进行赘述。在物理器件之上，我们有一层名为 Microarchitecture (微体系) 的结构层次，在这个层次中，底层的物理设备被组织为了功能性的数字单元，我在 数字电路与计算机架构基础 & 数字逻辑系统设计 系列文章的起始部分中花费了很多笔墨来描述这个层次是如何工作的，在这里我进行简要的回顾总结。在中央处理器中，我们可以简单的理解里面有用于存储数据的寄存器，以及用于计算的 ALU (arithmetic logic unit) 单元。在每个时钟周期内，若干个操作数 (operand) 会从寄存器里被读取出来送入 ALU 中，ALU 计算之后又会将计算结果送到寄存器中，我们将这样的流程称之为数据通路 (Data Path)。数据通路本质上是在执行一系列的存储在内存中的指令 (instruction)，我们可以将指令视为系统上层程序在最底层的表现形式，指令会控制数据在数据通路上的基本行为。一般来说，在一个硬件平台上，这些指令都是规范好的，多个指令构成了一个指令集，一个指令集基本上会 cover 数据在数据通路上的所有行为。硬件(寄存器、ALU等)和指令集都是直接呈现给汇编程序员的，因此我们就把硬件和指令集统称为 指令集体系结构 (Instruction Set Architecture, ISA)。

    通常来说，指令集会由 50~300 条指令组成，控制着所有数据在具体硬件上的行为，这是十分复杂和枯燥的。此时 操作系统 (Operating System) 的层次帮我们屏蔽了底层指令集的复杂性，即它帮上层应用程序屏蔽了底层硬件的复杂性、异构性以及异步性，向上呈现出了相对简单的系统逻辑概念 (进程、文件等)，这也是操作系统的功能意义之一。在操作系统之上，我们就会有基于这些相对简单的系统概念开发的各类应用程序，比如编译器、Shell解释器和 Web 程序等等。同时，我们希望操作系统能够帮助我们协调底层的硬件，以供上层多个应用程序使用。因此，在所有针对操作系统的讨论中，我们最终也就是想弄清楚两个问题：(1) 操作系统是如何屏蔽、虚拟化底层细节，并且 (2) 提供底层各类资源的管理、系统逻辑资源的调度的功能的，这也就是操作系统的两个基本功能。
