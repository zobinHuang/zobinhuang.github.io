1. Key-Value Storage

Log-Structured Merge Tree

  1. From WiscKey to Bourbon: A Learned Index for Log-Structured Merge Trees (OSDI'20)

    • Problem:

      在存储设备的读写带宽巨大提升的背景下，在传统的 LSM 存储中 (e.g. LevelDB, WiscKey) Indexing 在 performance breakdown 中占的比重越来越大 (i.e. 由 Indexing 和 Data Access 两部分组成)，因此作者想要通过借助 ML，使用 learning index 的方法来加速索引过程。

      对于之前的与 learning index 相关的工作来说，它更多的是对读过程进行优化，并且不允许原生数据集被破坏，否则需要对模型进行重新训练。然而，LSM 是一种专门为高性能写入而设计的 Key Value 数据库，频繁地写更新使得 learning index 的方法似乎变得不适用。

    • Solution:

      文章的作者发现，会导致较大 Indexing 延迟的，通常是在硬盘中已经被固化的、容量很大的 ssTable (i.e. in-memory 的 mmTable 并不会导致过大的索引延迟)。这些 ssTable 本身并不会被再次写入更新，并且各个 ssTable 以文件的形式存在，因此使用基于 learning index 的方法是可以在优化 LSM 存储查询索引过程的同时，避免重新训练模型的。

    • Remark:

      [1] 文章 Motivation 部分对 LevelDB 做了过程的详细介绍，并且还做了 performance breakdown；

分布式 Key Value 存储

  1. Pegasus: Tolerating Skewed Workloads in Distributed Storage with In-Network Coherence Directories (OSDI'20)

    • Problem:

      对于大规模的键值数据存储，为了应对单机存储容量的限制，更好的办法是分摊到多台服务器上进行存储。问题在于，在分摊之后，由于存在热点问题 (skewed object access pattern)，一小部分数据被读写的次数将远远超过其它数据。Selective Replicating 是一种有效的办法，将这些热点数据进行复制，以实现 load balance。在进行了复制之后，Consistency (一致性) 也必须进行解决。

      然而在大量的分布式数据面前，要定位哪些数据是热点数据是困难的。并且在分布式、高并发、高容量的情况下，保证 Consistency 也是相对困难的。

    • Solution:

      文章利用了 Programmable Dataplane 的 Switch 来路由前往各个数据服务器的请求。Switch 维护了一个目录表 (in-network coherence directory) 来记录哪些 objects 是被复制的，以及它们相应的位置，因此 Switch 能够对这些前往热点数据的请求进行负载均衡。由于处在主干位置，Switch 还能很好地对由于写操作引起的数据变更进行 Consistency 的保证。

    • Remark:

      [1] 文章 Motivation 部分对一个系统如何应对 skewed workload 进行了传统方法论的讨论，传统方法包括 Caching 和 Selective Replication。

基于 RDMA 的 In-memory Key Value 存储

  1. Fast RDMA-based Ordered Key-Value Store using Remote Learned Cache (OSDI'20)

    • Problem:

      in-memory 键值对存储通常使用 B+ 树等结构来构建索引，索引节点存储的是具体的 Value 的内存地址。

      [1] 传统的基于 RDMA 的 KVS 是让存储 Key-Value 对的 Server 来承担所有的请求，包括 GET、SCAN (static workload) 和 UPDATE、INSERT、DELETE (Dynamic Workload)。这样的方案虽然对网络的带宽消耗不是特别大，但是 Server 端的 CPU 却成为了性能瓶颈，在高速网络下随着 Client 个数的增加，系统性能将急剧下滑；

      [2] 后来有了 Client-centric 的 RDMA-based KVS，做法是对于 Static Workload 来说，让 Client 基于 RDMA 去访问 Server 端的索引，然后再对 Value 发起访问。但是这样会导致一次操作需要若干次网络操作，对网络的带宽产生了巨大的消耗；

      [3] 再后来有人尝试在 Client 端建立 Cache，以缓存索引查询结果，但是对于查询随机分布的需求来说，Cache 并不能解决问题

    • Solution:

      将 Machine Learning 引入 Client 端的 Cache。同样是针对 Static Workload，在 Client 端 Cache 做 ML 预测目标 key 在 Server 内存中的位置范围，然后只用一次 RDMA Read 将这个范围中的 index 都读到 Client，然后 Client 再基于读回的索引锁定目标 key 的具体位置，然后再次发起一次 RDMA Read 对目标 key 对应的 value 发起读取请求。这样就能实现 2 RDMA-Read/READ。

      为了避免需要重新训练模型，文章在 ML model 和 tree-based index 之间还做了一层 translation layer。ML model 输出的是一个逻辑地址，经过这个 translation layer 之后才是真正的物理地址。

    • Remark:

          None
