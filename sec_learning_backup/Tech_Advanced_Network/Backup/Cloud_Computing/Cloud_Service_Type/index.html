1. 常见的云计算服务模式

    常见的三种 云计算服务 分别是 SaaS (Software as a Service)，PaaS (Platform as a Service) 和 IaaS (Infrastrature as a Service)。结合上图，我们比较一下其区别：     

(1) SaaS (软件即服务)

    SaaS 即云服务提供商向用户直接提供应用程序，这些应用程序由第三方供应商直接管理，大多数为Web应用程序，用户不需要在客户端下进行任何的下载&安装即可使用，常见的例子如Google Gmail，Google Drive等；     

(2) PaaS (平台即服务)

    PaaS 即云服务提供商向用户提供云组件，大白话就是提供了一个软件创建平台，这个平台已经部署好了底层硬件、操作系统，还有可能另外还部署好了中间件、运行库等（这些都是可以定制的），用户可以在这个平台上直接创建自己的应用程序以及管理相应的数据内容。在实际应用中，PaaS大都是基于容器实现的，而我们下面要讲的 IaaS 大都是基于虚拟化技术实现的。容器的概念从上图在与虚拟化技术的比较中即可一目了然，即在一个主机操作系统下可以运行多个容器，这些容器互不干扰，资源占用都是动态的。容器是使用镜像构建的，其内部已经集成了运行/开发程序所需要的环境。PaaS的意义有几个：一是用户可以专注于应用程序逻辑的开发而无需关心实际的部署问题，使得整套流程变得更加简单，二是如果有多个开发人员涉及到云应用程序的开发，那么一个统一的程序开发环境将提供极大的便利。常见的例子有：AWS Elastic Beanstalk、Windows Azure、Heroku、Force.com、Google App Engine，Apache Stratos，OpenShift等；
    App Engine是PaaS模式的一种部署实现架构，App Engine将应用运行所需的 IT 资源和基础设施以服务的方式提供给用户，包括了中间件服务、资源管理服务、弹性调度服务、消息服务等多种服务形式。App Engine利用虚拟化与自动化技术实现快速搭建部署应用运行环境和动态调整应用运行时环境资源这两个目标。一方面实现即时部署以及快速回收，降低了环境搭建时间，避免了手工配置错误，快速重复搭建环境，及时回收资源， 减少了低利用率硬件资源的空置。另一方面，根据应用运行时的需求对应用环境进行动态调整，实现了应用平台的弹性扩展和自优化，减少了非高峰时硬件资源的空置。App Engine的架构如上图所示，各个部分的功能如下表列出所述。
组成模块说明

    简单提一下结合到的技术栈：
        (1) 底层依赖：对于资源隔离来说，其依赖的是Linux内核的名称空间 (namespace)，它提供了一种资源的隔离方案，即允许不同namespace下存在相同的PID，IPC和Network等系统资源，通过Namespace即可以进行区分；对于 资源控制来说，其依赖的是Linux内核的CGroup (control group)，CGroup本身是提供将进程进行分组化管理的功能和接口的基础结构，I/O或内存的分配控制等具体的资源管理功能是通过这个功能来实现的。CGroups可以限制、记录、隔离进程组所使用的物理资源（包括：CPU、memory、IO等），为容器实现虚拟化提供了基本保证。CGroups本质是内核附加在程序上的一系列钩子（hooks），通过程序运行时对资源的调度触发相应的钩子以达到资源追踪和限制的目的。
        (2) 应用程序载体：即容器的实现，业界目前靠的大都是 Docker，它可以把应用以及应用所依赖的环境完完整整地打成一个包，这个包拿到哪里都能原生运行。因此可以在开发、测试、运维中保证环境的一致性。Docker=LXC(Namespace+CGroups)+Docker Images，即在Linux内核的Namespace[资源隔离]和CGroups[资源控制]技术的基础上通过镜像管理机制来实现轻量化设计。
        (3) 容器集群管理：Kubernetes，它是一个由Google开源的容器集群管理系统。它构建Docker技术之上，为容器化的应用提供资源调度、部署运行、服务发现、扩容缩容等整一套功能。     

(3) IaaS (基础设施即服务)

    IaaS 即云服务提供商向用户提供可供自由组合的计算、存储、一定时间内的网络带宽、公网ip、以及一些增值服务（如监控、自动伸缩等）等资源，用户可以提出自定义的资源租赁需求，并通过 购买虚拟机 的方式来获取请求的主机资源，然后在其上可以运行自定义的操作系统、中间件从而开发自己的应用程序。IaaS的意义有几个：对于初创公司来说，IaaS使得他们不需要购买实际的硬件就能拥有满足自己需求的数据中心，对于快速发展的公司而言，IaaS具有极高的可扩展性和灵活性来支持他们不断扩张的业务。IaaS 又可以分为 私有IaaS 和 公有IaaS，前者由企业/组织在内部自己搭建，将私有云基础设施资源作为服务向公司里的各个部门提供，后者由专门的云服务提供商以租赁形式向外界提供，常见的 公有IaaS 例子有：DigitalOcean，Linode，Rackspace，AWS，Cisco Metapod，Microsoft Azure，Google Compute Engine（GCE）等；
    我们关注一下IaaS云端的基础架构，如上图所示，可以分为三层：
        (1) 云管理器 是云端对外的出口，在这里验证用户身份，管理用户权限，向合法用户发放票据（然后用户持此票据使用计算资源）、分配资源并管理用户租赁的资源。与之相关联的存储是 DOS (Data Object Storage)，负责存储与用户相关的元数据。
        (2) 集群管理器 负责管理本集群内部的高速互联在一起的计算机，一个集群内的计算机可能有成百上千台。集群管理器接受上层的资源查询请求，然后向下层的计算机管理器发送查询请求，最后汇总并判断是部分满足还是全部满足上层请求的资源，再反馈给上层。如果接下来收到上层分配资源的命令，那么集群管理器指导下层的计算机管理器进行资源分配并配置虚拟网络，以便能让用户后续访问。与之相关联的存储是 PLS (Persistent Local Storage)，用于存储本集群内的全部虚拟机镜像文件，这样一台虚拟机就能在集群内任意一台计算机上运行，并轻松实现虚拟机热迁移。
        (3) 计算机管理器 (CM) 在每台计算机上都有搭载，它一方面与上层的集群管理器打交道，另一方面与本机上的虚拟机软件打交道。它把本机的状态（如正在运行的虚拟机数、可用的资源数等）反馈给上层，当收到上层的命令时，计算机管理器就指导本机的虚拟机软件执行相应命令。这些命令包括启动、关闭、重启、挂起、迁移和重配置虚拟机，以及设置虚拟网络等。

2. 总结

    在了解完云服务提供商提供的服务类型后，我们可以总结出云服务提供商提供云服务的基本思路。如上图所示，我们通过系统的结构来明确云服务供应需要考虑的东西。
    首先我们必须明确我们提供的是哪一个层次的云服务。如果我们是对企业/个体用户直接提供Web/移动端App等应用的话（分别对应ToB，ToC），我们就是在做SaaS；如果我们不直接提供可以直接使用的应用程序，而是提供在线计算(e.g. Apache Flink, Spark)、离线计算(e.g. yarn)，在线存储(e.g. ElasticSearch, MySQL, Redis, Kafka)、离线存储(e.g. HDFS, Hive)、监控告警、源数据(e.g. DNS, Zookeeper)等可用于构建应用程序的在线/离线组件，我们就是在做PaaS；如果我们是将底层计算、存储和网络资源打包封装租赁给其他用户或者提供给自己企业内部的其它业务，那么我们就是在做IaaS。
    明确了我们在做的东西是哪个层次的云服务以后，给我们的云服务提供支持的底层服务就可以千变万化了。比如，在构建SaaS的顶层应用的时候，我们既可以选择基于PaaS平台提供的接口来创建应用，也可以选择基于IaaS提供的平台自己搭建。而对于一个全面的能提供所有三个层次的云服务提供商来说，PaaS和IaaS的选择也是自由的，既可以选择自家的，也可以基于其他云平台提供的这两种服务来实现上层需求，这就与具体的需求、成本和预期投入有关系了。
