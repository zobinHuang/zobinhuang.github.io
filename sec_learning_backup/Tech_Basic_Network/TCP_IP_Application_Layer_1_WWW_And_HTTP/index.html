1. 万维网体系结构

    这一小节用于理清楚一些概念。万维网 (World Wide Web, WWW) 是分布式的 Client-Server 模式服务。服务的提供者分布在被称为 网站 (site) 的地方提供服务。每一个网站保存一个或多个文档，被称为 Web 文档。Web 文档可以承载文字、图片&视频 (超媒体, Hypermedia)，以及前往其他 Web 文档的链接 (超文本, Hypertext)。
    由于是 Client-Server 模式的服务，Web 服务中又可以区分出两类实体：Web 客户 和 Web 服务器。Web 客户 就是我们熟知的浏览器，它接受来自用户的输入，向服务端发送相应请求，并将获得的 Web 文档 渲染在浏览器 UI 上。 Web 服务端 用于存储 Web 文档，接受来自客户端的请求并做相应处理。
    因为 Web 客户端需要对目标 Web 文档资源进行请求，所以我们需要一种能对 Web 文档进行定位的方法。统一资源定位符 (URL) 就是在 Internet 上对任何种类信息进行定位的标准，标准格式如下所示：

协议 :// 主机 : 端口 / 路径

    协议 是指 Web 客户端 和 Web 服务端 之间交互所使用的协议标准，如 HTTP, HTTPS, FTP, TELNET等
    主机 是指通常使用域名定位的计算机网络地址。
    端口 是可选择性地出现在 URL 中的。
    路径 是指 Web 文档 在 Web 服务端存放的系统路径名，

2. Web 文档类型

(1) 静态文档

    静态 Web 文档在服务器中被创建，并且其文件内容是在文件被创建时确定的，而非在客户端在请求时确定的。客户端请求静态 Web 文档时获取的是该静态 Web 文档的一份拷贝。常见的用于准备静态文档的语言有诸如 HTML, XML, XSL 和 XHTML 等。

(2) 动态文档

    动态 Web 文档是在 Web 客户端 请求时才被创建的。请求到达时，Web 服务端就会运行创建动态文档的应用程序或脚本，并且把程序/脚本的输出作为对请求该 Web 文档的 Web 客户端 的回应。
    这里简单穿插一下关于 Web 服务器创建动态文档的历史[1, 2]。
    时间回到 1993 年，通用网关接口 (CGI) 标准化了根据 Web 客户端请求生成动态网页的标准，其标准化了服务端的输入、处理过程和输出过程。CGI 输入即 Web 客户端 的请求，Web 客户端有针对性的请求是使用 表单 (form) 来实现的。如果表单很短，可以使用接在 URL 中 '?' 后面的形式发送给服务器，如 "http://www.deanza/cgi-bin/prog.pl?23"，其中的 23 就是作为表单内容。如果表单内容较长，则 Web 客户端可以请求服务器发送表单，Web 客户端将用户输入数据填入该表单并发送给服务器作为 CGI 程序的输入。CGI 输出即值的是 CGI 程序的输出，通常是普通正文或 HTML 结构的正文，但也可以是其它形式。
    然而，在当时的 CGI 实现中，针对每一条请求，服务端都得单独开辟一个进程用于处理请求，十分耗费服务器的 CPU 和 内存资源。而且动态创建的文档中的部分内容每次都是固定的，每次都使用 CGI 脚本重新生成十分浪费资源，因此到了 1995 年，PHP、ColdFusion 和 ASP (Active Server Pages) 等相继出现用于改善服务端生成动态 Web 文档的性能。

(3) 活动文档

对于某些应用，我们需要程序能在客户端运行，比如 网页游戏、在线 Word 文档编辑、前端表单验证等功能，此时根据 Web 客户端的请求，Web 服务端会将相应的程序文件 (e.g. Java, Flash) 或脚本文件 (e.g. JavaScript) 发回给 Web 客户端执行。

3. HTTP (超文本传送协议)

    HTTP 是用于在 Web 客户端 和 Web 服务器之间用于存取数据的应用层协议，其传输层基于 TCP 协议 (端口: 80)。在 HTTP 1.1 以前，HTTP 默认使用的是非持续连接，即针对每一次 HTTP事务 (见下面) 都需要进行一次 TCP 建链和拆链，在 HTTP 1.1中，则变为了持续连接，即 Web 客户端 建立一次连接后，Web 服务端 会将连接保持到 超时 或者 Web 客户端 请求关闭连接为止。

(1) HTTP 事务

    虽然 HTTP 基于 TCP，但是它是一种无状态的应用层协议，即 Web服务端 不会保存任何和 Web用户端 有关的状态信息，Web 服务端 专心地对 Web 客户端 的请求作出具体响应。Web 客户端请求和 Web 服务端 响应的这样的过程被称作一个 HTTP 事务，如上图所示。     

    (a) 请求报文

    上图给出了一个 HTTP 请求报文的例子，我们将一个 HTTP 请求报文分为三个部分：请求行 (Request Line)、首部行 (Request Header) 和 主体(Entity)，下面分别作阐述。

请求行 (Request Line)

方法 space (空格) URL space (空格) 版本 /r (回车) /n (换行)

    在请求行中，我们有 方法、URL 和 版本 三个字段。URL 定义了 Web 文档的具体网络地址，版本 给出了 HTTP 协议的版本号，方法 字段定义了请求的类型，请求的类型有若干种，具体如下所示：

OPTIONS 询问关于可用的选项

主体 (Entity)

    主体会出现在某些 HTTP 请求报文中，用于添加一些需要发送的备注信息。

    (b) 响应报文

    上图给出了一个 HTTP 响应报文的例子，我们将一个 HTTP 响应报文分为三个部分：状态行 (State Line)、首部行 (Request Header) 和 主体(Entity)，下面分别作阐述。

状态行 (State Line)

版本 space (空格) 状态码 space (空格) 短语 /r (回车) /n (换行)

    状态行共有三个字段。版本 用于描述 HTML 版本信息，状态码 和 状态短语 用于描述请求的状态，100 系列的代码代表着 Web 服务端提供的一些信息，200 系列的代码代表成功的请求， 300 系列的代码代表把客户重新定向到另一个 URL，400 系列的代码代表在客户端的差错，500 系列的代码代表在服务端的差错，具体的常见状态码和状态短语如下所示：

提供信息的

101 Switching 服务器同意切换协议
成功
200 OK 请求成功
201 Created 新的 URL 被创建
202 Accepted 请求被接受，但还没有马上起作用
204 No Content 主体中没有内容
重新定向
301 Moved Permanently 服务器已不再使用所请求的 URL
302 Moved Temporarily 所请求的 URL 已暂时地删除了
304 Not Modified 文档还没有被修改
客户差错
400 Bad Request 在请求中有语法差错
401 Unauthorized 请求缺少适当的授权
403 Forbidden 服务被拒绝
404 Not Found 文档未找到
405 Method Not Allowed 这个方法不被这个 URL 所支持
406 Not Acceptable 所请求的格式不可接受
服务差错
500 Internal Server Error 服务端有差错，可能是崩溃了
501 Not Implemented 所请求的动作不能完成
502 Service Unavailable 服务暂时不可用

首部名称 : space (空格) 值 /r (回车) /n (换行)

首部 描述

Content-Encoding 指明编码方案
Content-Language 指明语言

主体 (Entity)

    HTTP 响应报文中包含的主体就是服务器向客户发送的文档 (如果处理没有发生错误的话)。

(2) Cookie

    前面讲到 HTTP 是一种无状态的应用层协议，但是考虑以下场景：
        (a) 某些网站需要之前用户登录过才能访问
        (b) 早期的网购网站在不依靠数据库的情况下，允许用户在网页上向购物车添加商品，后进行汇总支付
        (c) 广告公司想收集点开投放有广告的网页的用户的信息
    很明显我们需要一种能够记录 Web客户端 状态的办法，Cookie 就是用于这样的场景下。下面简单介绍一下 Cookie 的工作原理。

    Cookie 的创建和存储

    当 Web 服务端 收到 Web 客户端 的请求后，它会把与 Web 客户端 有关的信息存储在一个文件或者字符串内。当 Web 服务端 向 Web 客户端 响应请求时，它会把这个 Cookie 封装在响应报文中。当 Web 客户端 收到响应时，浏览器就会把这个 Cookie 存储在 Cookie 目录中，浏览器的 Cookie 目录通常是按照域名来对 Cookie 进行分类的。

    Cookie 的使用

    当一个 Web 客户端 向 Web 服务端 发送请求时，浏览器就查找在 Cookie 目录中是否有那个服务器发送的 Cookie。如果找到了就把这个 Cookie 包含在请求中。当 Web 服务端 收到这个请求时，它就知道这是一个老客户而不是新客户。

(3) Web 缓存：代理服务器 (Proxy Server)

    HTTP 代理服务器 用于缓存 Web客户端 最近请求的响应的副本。代理服务器减小了原始服务器的负荷，减小了通信量，也减小了延时。代理服务器同时扮演着 Web客户端 和 Web服务端 的角色。当它从 Web客户端 接收到 HTTP 请求并发现有对应的响应存在时，它就作为 Web服务端 响应 Web客户端 的请求，否则它就以 Web客户端 的身份向远端 Web服务器 发起请求，接收到响应后再作为 Web服务端 将响应发送给客户。
