    我们本节将以 intel 的 igb 驱动为例子来观察 NAPI 系统的工作流程。

    首先，我们来看 igb 驱动注册的 ISR，如下所示，它是在 drivers/net/ethernet/intel/igb/igb_main.c 中定义的。

    可以很主观的看出，ISR 里面要做的工作必须保证少且快速，以实现对 Interrupt 的快速响应。其中我们重点要关注的是对 napi_schedule 的调用，它是在 include/linux/netdevice.h 中定义的。它实际上调用了 __napi_schedule，后者是在 net/core/dev.c 中定义的，而 __napi_schedule 实际上则调用了 ____napi_schedule，它也是在 net/core/dev.c 中被定义的。它们的定义如下所示。

    可以看到，首先，在 napi_schedule 中，我们首先调用 napi_schedule_prep 来判断当前处理的 napi_struct 是否正在被轮询处理中，如果该函数返回为 0，那么就说明正在被处理中，理论上本不应该产生中断，可是却依然产生了，我们在这里相当于挡了一下意外情况。如果当前的 napi_struct 结构没有正在被轮询处理，则会进而调用 __napi_schedule 函数，它在一个关闭了对 Interrupt 的响应的条件下，又调用了 ____napi_schedule，注意到此时传入给后者的参数中，一个是 per-cpu 的 softnet_data 结构，一个是当前正在处理的 napi_struct。在 ____napi_schedule 中，我们会调用 list_add_tail 来把当前的 mapi_struct 的 poll_list 成员连接到当前 CPU 下的 softnet_data 的 poll_list 成员下，并且调用 __raise_softirq_irqoff 来拉高 bitmask __softirq_pending 中的 NET_RX_SOFTIRQ 标志，以标识该 Softirq 进入 Pending 状态。

    对应于 Softirq NET_RX_SOFTIRQ 的处理函数是 net_rx_action，它是在 net/core/dev.c 中定义的，其定义如下所示:

    回顾我们在 Softirq 中讨论 Softirq 时，分析的 __do_softirq 函数，由于 Softirq 是在关闭了抢占的环境下执行的，因此其通过控制自己重复检查 bitmask 以及运行时间控制了自己的迭代次数，以保证其他进程不会因为 Softirq 的运行而被延后过久。同样的，在 net_rx_action 中，我们可以观察到类似的代码，net_rx_action 通过 budget 来控制自己的循环次数，每一次对 napi_poll 的调用都会扣除掉一定的数值。当该数值小于 0 之后，或者总的运行时间超过一定额度之后，net_rx_action 就会终止自己的循环，这样的做法保证了 Kernel 中其他 Softirq 的处理时间。我们后面在对各个 device 的 poll 函数的分析中还会看到，每个 Device 在自己的 poll 函数中也会严格控制自己的循环次数，以保证对各个 Device 的轮询是公平的。
